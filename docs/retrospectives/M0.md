## Sub-deliverables

### D1 — Initial repository and Django foundation

**Scope:** Backend skeleton, settings modules, Docker Compose, Django Dockerfile,
Nginx config, custom User model in migration #1, app skeletons, custom landing
page, login scaffold, platform-console placeholder, health endpoints, Celery
app, pytest config, ruff/mypy config, CI scaffold, user-model baseline check.

**Verification:**

- `make lint` — ruff clean, mypy clean, no dead suppressions
- `make test` — 14 passed
- `make check` — baseline PASS, django-check clean
- Manual: `/`, `/login/`, `/platform/`, `/healthz`, `/readyz`, `/django-admin/`,
  Mailpit at `:8025`, MinIO at `:9001` all reachable
- allauth deprecation warnings cleared (ACCOUNT_LOGIN_METHODS + ACCOUNT_SIGNUP_FIELDS)

**Decisions:**

- Python 3.14 base image targeted. Dockerfile includes build-essential as a
  fallback for transitive deps that don't yet ship cp314 wheels.
- Tailwind browser CDN retained in `base.html` for D1; replaced by the
  compiled Vite pipeline in D3.
- mypy in CI was non-blocking during D1; promoted to blocking at the close of D1.

### D2 — Platform models + seed_v1

**Scope:** Organization, Membership, TenantExportRequest, TenantDeletionRequest;
Capability, Role, RoleCapability, MembershipRole, MembershipCapabilityGrant;
`seed_v1` data migration with 96 capabilities, 11 default role templates, and
the System User row; dev-only Django admin for raw inspection.

**Verification:**

- `apps/platform/rbac/tests/seeds/test_seed_v1.py` — all assertions pass:
  exactly one System User, 96 capabilities, 11 default role templates with
  expected capability sets, idempotency under re-run.
- Migration chain verified: `platform_accounts.0001` → `platform_organizations.0001`
  → `platform_rbac.0001` → `platform_rbac.0002_seed_v1`.
- `/django-admin/` shows Organization, Membership, Capability, Role,
  RoleCapability, MembershipRole, MembershipCapabilityGrant for raw inspection.

**Decisions:**

- Owner and Org Admin templates hold identical capability sets in v1.
- `Organization.default_tax_jurisdiction_id` and `invoicing_policy_id` are
  plain UUID columns; convert to real FKs when TaxJurisdiction (M3) and
  InvoicingPolicy (M5) land.
- `Organization.accounting_adapter_config` ships as plain JSONB; replaced
  with `EncryptedJSONField` when the first concrete adapter ships (M5).
- `MembershipScopeAssignment` deferred until apps.operations.locations lands.
- D2 follow-up applied in D3: Sales Staff template now includes
  `clients.contacts.manage` and `clients.locations.manage`. In-place edit to
  `v1_default_roles.py` is the last permitted in-place edit; future changes
  go through successor seed migrations per I.6.5.

### D3 — Frontend tooling + AST static check

**Scope:** Vite + **Tailwind 4** + TypeScript scaffold under `frontend/`;
django-vite wired into `config/settings/base.py` and `templates/base.html`;
Vite Compose service with named-volume install pattern; Nginx `/vite/` proxy
with runtime DNS resolution + WebSocket upgrade; service-layer discipline
AST check (advisory) at `scripts/check_service_layer_discipline.py`; CI
`service-discipline` and `frontend-build` jobs.

**Verification:**

- `make up` brings up Vite alongside web/worker/beat; the vite container
  performs its own `npm install` into the `vite-node-modules` named volume.
- `curl -sf http://mph.local/vite/@vite/client` returns HTTP 200.
- Manual HMR check: editing `frontend/src/styles/login_landing.css`
  updates the browser without a full reload.
- `make lint-service-discipline` runs on host; emits warnings only and
  exits 0 (advisory in M0 per A.4.5).
- Tailwind browser CDN removed from `templates/base.html`; replaced by
  `{% vite_hmr_client %}` + `{% vite_asset 'src/entries/<entry>.ts' %}`.

**Decisions:**

- **Tailwind 4** selected (guide H.1.3 originally specified 3.x; user
  authorized the deviation). Implementation uses `@tailwindcss/vite`
  plugin (no PostCSS pipeline; Lightning CSS handles compilation and
  prefixing).
- `@tailwindcss/container-queries` dropped — built into Tailwind 4.
- **`tailwind.config.cjs` retained** per H.2.1's "tokens defined ONCE"
  requirement, wired via `@config` directive in `_base.css`.
- **Vite HMR routed through Nginx** (`/vite/` proxy with WebSocket upgrade
  - runtime DNS resolution) so the dev experience stays on one hostname.
- **Named-volume install pattern for `node_modules`** — instead of host
  install + bind mount (which had cross-platform native-binary issues and
  Windows file-lock issues), `node_modules` lives in a Docker named volume
  (`vite-node-modules`). The container runs `npm install` on first start.
  Trade-off: editor IntelliSense on `frontend/` won't see `node_modules`
  unless the engineer also runs a host install. We'll revisit if the
  ergonomics bite.
- **Node 24 in container** matches realistic host versions.
- Service-discipline check stays advisory in M0; promotion to blocking
  is scheduled for M2 per A.4.5. Uses Python's stdlib `ast` module.
- Email-template-preview bundle ships now even though we don't yet have
  email-preview screens — Phase 1 invoice/notification HTML templates
  land in M5/M7 and we want the entry pre-wired.

**Decisions:**

- `seed_dev_tenant` does direct ORM `.create()` calls because the service
  layer (`services.create_organization`) lands in M1. The AST check
  exempts `management/commands/` paths alongside admin/migrations/tests.
- The command refuses to run unless `DJANGO_SETTINGS_MODULE` is
  `config.settings.dev` or `config.settings.test`, or `DJANGO_DEBUG=True`.
- `--reset` deletes Org-scoped tenant data and the admin user (only if
  the admin has no other memberships). The System User and the seed_v1
  template Roles + Capabilities are never touched.
- Default admin password is `mph-demo-password!` — printed in the
  command output for engineer convenience. Override with `--admin-password`.
- `dev-setup.sh` is bash-only (Linux/macOS). Windows engineers use the
  underlying `make` targets directly; README documents this.

## M0 closeout

M0 is the foundation milestone. Everything in this repo, every milestone
that follows, presumes:

- A custom User model.
- A platform-tier RBAC capability registry with the 96 v1 codes already
  in the database.
- 11 default role templates with `organization=NULL`.
- A System User row whose `is_system=True` invariant is database-checked.
- A row-level multi-tenancy posture, with tenant-owned models still to
  be defined in M1 against the `TenantOwnedModel` / `TenantManager` /
  `TenantQuerySet` primitives.
- A Vite + Tailwind 4 + HTMX frontend pipeline that produces compiled
  assets at build time and serves HMR through Nginx in dev.
- A service-layer discipline AST check that fires (advisory) on any
  state mutation outside `apps/*/services/`.
- CI gates: ruff, mypy (blocking), pytest, frontend-build (blocking),
  and service-discipline (advisory).

M1 picks up from here with the tenancy primitives.
