**Deviations from the guide:**

1. **B.5.4 dual-tier lockout NOT implemented.** Guide specifies
   "5 failed in 15 min ‚Üí 15 min lock; 10 failed in 1 hour ‚Üí 24 hour
   lock + `ACCOUNT_LOCKED`". Allauth provides only a single-tier
   lockout (`ACCOUNT_LOGIN_ATTEMPTS_LIMIT` + `_TIMEOUT`); the second
   tier requires a custom rate-limiter. **Tracked as a follow-up
   alongside M2 audit work.** The single-tier lockout we have is
   set to 5 attempts / 15-minute timeout. `ACCOUNT_LOCKED` event is
   registered in `_KNOWN_EVENT_TYPES` but not yet emitted.

2. **B.5.10 rate limits mapped to allauth vocabulary.** The guide
   specifies per-IP and per-email limits as separate buckets; allauth
   provides per-action limits with a combined "n per minute, n per
   hour" syntax. The closest mapping is in `ACCOUNT_RATE_LIMITS`.
   Per-IP+per-email split requires django-ratelimit, deferred.

3. **`RequireMfaEnrollmentMiddleware` is custom.** allauth.mfa ships
   a *challenge* middleware (intercept post-login when user has TOTP)
   but does NOT ship a *forced enrollment* middleware. We built our
   own because B.4.9 / H.3.4 require both. This is the only piece of
   bespoke auth-flow code in M1 D4.

**Bugs caught and fixed during development:**

* `'User' object has no attribute 'username'` ‚Äî allauth's
  `default_user_display` reads `user.username`. Fix:
  `ACCOUNT_USER_DISPLAY` settings hook ‚Üí `user_display.py` callable.
  Test: `test_user_display.py`. The smoke suite would have caught
  this in CI had it existed at the time.
* `'counter' argument to 'blocktrans' must be a number` ‚Äî assumed
  allauth exposes `unused_count` to the recovery-codes template; it
  does not. Fix: `{% with unused_count=unused_codes|length %}` in the
  template. The smoke suite would have caught this in CI; it now
  does.
* `select-org/` was reachable post-login despite the user having no
  TOTP enrollment. Allauth.mfa has no forced-enrollment middleware.
  Fix: `RequireMfaEnrollmentMiddleware`.
* `seed_dev_tenant`'s demo admin had no `EmailAddress` row, so
  `ACCOUNT_EMAIL_VERIFICATION = "mandatory"` blocked dev login. Fix:
  command now creates a verified primary `EmailAddress` via
  `update_or_create`.
